## 1. How did you divide the work between you and your partner?
_Our team divided the work by assigning ownership of specific features and tasks to each person. Instead of each  of us working on the same part at the same time, we first identified the required features, then “dibs” each one based on who was available, confident, or better prepared to handle it. This approach helped us avoid duplicated effort and made it easier to stay organized while working on separate branches. At the same time, we still communicated regularly to update each other on progress, clarify dependencies, and make sure our individual work would still fit together as one final project. Overall, dividing the work by feature gave us a practical structure and allowed us to contribute more efficiently as a team._

## 2. What Git strategies or commands helped you most during the project?
_The Git commands that helped us the most were git fetch and git pull. Since we were working across multiple branches, it was important for us to stay updated on what changes had already been made by others and what new updates were available in the remote repository. git fetch was especially useful because it allowed us to check for updates from the remote server without immediately changing our local files, which gave us a chance to review the state of the project first. On the other hand, git pull helped us bring the latest changes from the remote branch into our local branch so we could continue working with the most updated version. These commands were important in keeping our work synchronized, reducing confusion, and helping us avoid larger conflicts later in the project._

## 3. Describe a merge conflict you encountered. What caused it and how did you resolve it?
_ne merge conflict we encountered happened when changes were made to the same file on different branches. This usually occurred when both of us edited related parts of a file or when one branch had already updated content that the other branch had not yet pulled. As a result, Git could not automatically determine which version should be kept during the merge. To resolve the issue, we first checked the conflicting file carefully and reviewed the marked sections that Git identified. We then discussed which changes were correct, compared the intention behind each edit, and manually combined the necessary parts before saving the file. After that, we staged the resolved file and committed the merge. One important lesson we learned is that frequent pulling, clearer task boundaries, and regular communication can significantly reduce the chances of merge conflicts and make them easier to solve when they happen._

## 4. What were the biggest challenges you faced as a team?
_One of the biggest challenges we faced as a team was figuring out where to begin and how to move forward in a clear and organized way. At the start of the project, we spent a lot of time discussing possible approaches, which was helpful for brainstorming but also made it difficult to decide on one direction immediately. Because there were several possible solutions, we sometimes felt overwhelmed and unsure about the next step. Another challenge was balancing collaboration with execution, since too much discussion could slow us down, while moving too quickly without alignment could create confusion later. Despite these difficulties, the discussions were still valuable because they helped us explore better ideas, understand the task more deeply, and eventually settle on workable solutions together._

## 5. What did you learn about using Git in a collaborative setting?
_Using Git in a collaborative setting taught us that version control is both powerful and demanding. It is exciting because it allows multiple people to work on the same project at the same time, track changes, create separate branches, and combine work in a structured way. However, it can also be stressful because even small mistakes, such as forgetting to pull recent changes or working on the wrong branch, can affect the workflow of the whole team. This experience showed me that Git is not just about knowing commands, but also about practicing discipline, communication, and awareness of the project’s shared state. I learned that successful collaboration with Git depends on both technical knowledge and good teamwork habits._

## 6. How would you improve your workflow next time?
_If we were to do this project again, we would improve our workflow by creating a more concrete plan from the start and setting aside more time for discussion, coordination, and studying the exercise before jumping into implementation. A clearer division of tasks, agreed timelines, and a shared understanding of the workflow would help us work more confidently and avoid unnecessary delays. I would also make it a habit to check branch status more often, pull updates regularly, and confirm responsibilities before editing shared files. These small improvements would likely make our process smoother, reduce confusion, and help us collaborate more effectively as a team._
